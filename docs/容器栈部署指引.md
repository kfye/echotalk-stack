# EchoTalk 容器栈部署指引（Day 1）

> 对应 14 天冲刺计划 Day 1：「服务器用 Portainer 起栈：Nginx + MySQL 8 + Redis + Go 占位服务」。
> 前提：Portainer 已部署。本文给出栈说明 + Portainer 操作步骤 + 验收方法。

## 1. 这套栈是什么

`echotalk-stack` 是**基建/部署仓**（区别于 backend / app / admin 三个业务仓）。它用一份 `docker-compose.yml` 在单台 **2核2G** 服务器上拉起四个容器，全部由 Portainer 统一管理：

| 容器 | 镜像 | 作用 | 端口 | 内存上限 |
|---|---|---|---|---|
| `echotalk-nginx` | nginx:1.27-alpine | 反向代理 + 托管管理端静态文件 | **80（公网）** | 64M |
| `echotalk-app` | 本地构建（Go 占位） | 占位业务服务，暴露 `/health` 并探测 DB/Redis | 仅内网 8080 | 128M |
| `echotalk-mysql` | mysql:8.0 | 关系库（已调瘦） | **仅内网 3306** | 512M |
| `echotalk-redis` | redis:7.2-alpine | 缓存（限 128M） | **仅内网 6379** | 200M |

合计内存上限约 **0.9G**，加上 OS + Portainer，留给 2G 机器足够余量（仍建议配 2–4G swap，见 §6）。

### 关键设计（贴合技术方案 §9）
- **MySQL/Redis 绝不暴露公网**：compose 里用 `expose` 而非 `ports`，只在 `echotalk` 内网可达；服务器安全组只放 80/443。
- **每容器设 `mem_limit`**：互不抢占，防止小内存机 OOM。
- **MySQL 调瘦**：`innodb_buffer_pool_size=128M`、关 `performance_schema`、`max_connections=50`（见 [mysql/conf.d/echotalk.cnf](../mysql/conf.d/echotalk.cnf)）。
- **Redis 限内存**：`maxmemory 128mb` + `allkeys-lru`（见 [redis/redis.conf](../redis/redis.conf)）。
- **App 经 healthcheck 编排**：Nginx 等 app 健康、app 等 MySQL/Redis 健康后才起，依赖顺序正确。

### 目录结构
```
echotalk-stack/
├── docker-compose.yml        # 栈定义
├── .env.example              # 环境变量样例（复制为 .env / 填到 Portainer）
├── nginx/
│   ├── nginx.conf            # 主配置（上传体积、gzip）
│   └── conf.d/echotalk.conf  # 反代 /api → app，/ → 管理端静态，含 HTTPS 模板
├── mysql/conf.d/echotalk.cnf # MySQL 调瘦
├── redis/redis.conf          # Redis 限内存
├── admin/dist/               # 管理端构建产物（现为占位页，Day 6 替换）
└── placeholder/              # Go 占位服务（Day 2 起由真实 backend 替换）
    ├── main.go  go.mod  Dockerfile
```

## 2. 部署方式选择

Portainer 起栈有两种，**推荐方式 A（Git 仓库）**，改配置只需 push + 重新拉取，干净可追溯：

- **A. Repository（推荐）**：Portainer 直接从 Git 仓库拉 `docker-compose.yml` 部署。
- **B. Web editor**：把 `docker-compose.yml` 内容贴进 Portainer 编辑器。简单但配置文件（nginx/mysql/redis conf）需另想办法（见 §4 注意）。

> ⚠️ 本栈用到了仓库内的多个挂载文件（nginx/mysql/redis 配置、admin 静态、placeholder 构建上下文）。**方式 B 纯网页粘贴会缺这些文件**，所以强烈建议用方式 A，或先把整个仓库 `git clone` 到服务器再用「本地路径」部署。

## 3. 操作步骤 —— 方式 A（Git 仓库，推荐）

1. **把本仓库推到 Git 远程**（GitHub / Gitee / 自建 GitLab 均可），确保包含上面所有文件。
2. Portainer 左侧 **Stacks → + Add stack**。
3. **Name**：`echotalk`。
4. **Build method** 选 **Repository**：
   - Repository URL：你的仓库地址
   - Reference：`refs/heads/main`
   - Compose path：`docker-compose.yml`
   - 私有仓库则打开 Authentication 填账号/Token。
5. **Environment variables** 区域逐条填入（对应 `.env.example`）：
   - `MYSQL_ROOT_PASSWORD`、`MYSQL_DATABASE`（建议 `echotalk`）、`MYSQL_USER`、`MYSQL_PASSWORD`
   - 用强密码；这些不要写进仓库。
6. 点 **Deploy the stack**。Portainer 会构建 `app` 镜像并拉起 4 个容器（首次构建 Go 镜像约 1–2 分钟）。

## 4. 操作步骤 —— 方式 B（服务器本地 + Web editor 备选）

如果不想接 Git，可在服务器上：
```bash
git clone <仓库地址> /opt/echotalk-stack
cd /opt/echotalk-stack
cp .env.example .env && vi .env   # 填密码
docker compose up -d --build
```
然后 Portainer → Stacks 会自动发现这个由 CLI 创建的栈（Portainer 可纳管 external stack）。这样既有完整文件，又能在 Portainer 里观察/重启。

## 5. 验收（Day 1 完成标准）

逐项核对计划里的 DoD：

1. **Portainer 见 4 容器健康**
   Portainer → Containers，应看到 `echotalk-nginx / app / mysql / redis` 均 `running`，且 app/mysql/redis 状态带 `(healthy)`。

2. **`GET /health` 返回 200**
   ```bash
   curl -i http://<服务器IP>/health
   ```
   预期 200，body 形如：
   ```json
   {"status":"ok","time":"2026-06-24T...","deps":{"mysql":true,"redis":true}}
   ```
   `deps` 两项都为 `true` 即证明 **app 能连通 MySQL 与 Redis**（计划要求的「能连 MySQL/Redis」）。

3. **MySQL/Redis 未暴露公网**（安全核验）
   ```bash
   # 从服务器外部扫这两个端口应不可达：
   nc -vz <服务器IP> 3306    # 预期 失败/超时
   nc -vz <服务器IP> 6379    # 预期 失败/超时
   ```
   仅 80 可通。

4. **管理端静态托管可访问**
   浏览器开 `http://<服务器IP>/` 看到占位页（Day 6 换成 Vue 产物）。

> 说明：本占位服务用的是 **TCP 连通性探测**，不建表。计划里「GORM 自动建表成功」属于 **backend 仓**的 Day 1 脚手架任务（gin+viper+GORM+zap），在真实服务替换 `placeholder/` 后完成；本栈先保证基础设施与连通性就位。

## 6. 上线/加固清单（Day 13 收口，先知道）

- [ ] **Swap**：`fallocate -l 2G /swapfile && chmod 600 /swapfile && mkswap /swapfile && swapon /swapfile`，并写入 `/etc/fstab`。小内存机救命缓冲。
- [ ] **重启自启**：本栈所有容器已设 `restart: unless-stopped`，服务器重启后自动拉起。
- [ ] **备份**：定时 `mysqldump` → 上传 COS，保留最近 N 天（计划 Day 13）。
- [ ] **HTTPS**：域名备案 + 证书（腾讯云免费 SSL / Let's Encrypt）后，按 [nginx/conf.d/echotalk.conf](../nginx/conf.d/echotalk.conf) 文末模板开启 443，并在 compose 的 nginx `ports` 加 `"443:443"`、挂载证书目录。
- [ ] **Redis 密码**（可选）：解开 `redis/redis.conf` 的 `requirepass`，并在 app 配置里带上。

## 7. 常用运维命令

```bash
# 查看栈状态 / 日志
docker compose ps
docker compose logs -f app
docker compose logs -f mysql

# 改了配置后重建并重启
docker compose up -d --build

# 进 MySQL（容器内，仅内网）
docker exec -it echotalk-mysql mysql -uroot -p

# 进 Redis
docker exec -it echotalk-redis redis-cli

# 停止/删除栈（数据卷保留，除非加 -v）
docker compose down
```

## 8. 后续衔接

- **Day 2**：用真实 `backend` 仓替换 `placeholder/`。两种接法：① 把 backend 的镜像 push 到 registry，compose 里 `app.image` 指向它、去掉 `build`；② 或在 compose 里把 `build.context` 指向 backend 仓路径。真实服务暴露同样的 `:8080` 与 `/health` 即可无缝替换，Nginx 反代无需改。
- **Day 6**：`admin` 仓 `npm run build` 产物拷到 `admin/dist/`（或在 CI 里同步），Nginx 自动托管。
